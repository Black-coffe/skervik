## 1. Цели и нефункциональные требования

### 1.1. Продуктовые цели
1. Партия укладывается в **≤60 минут** (синхронный онлайн); адаптивная длительность.
2. **Честный и прозрачный** ГСЧ — проверяемый сид (закрываем главную боль конкурентов).
3. **Catch-up механики** против runaway leader / kingmaking (профили правил).
4. Бесшовный **мультиплеер 2–10**, single player с ботами, async-режим.
5. **Кросс-платформа**: web-first (PWA), один аккаунт везде.
6. Open-source: воспроизводимая сборка, понятная архитектура, низкий порог контрибьюции.

### 1.2. Нефункциональные требования (NFR)

| Категория | Требование | Целевое значение (старт) | Целевое значение (масштаб) |
|---|---|---|---|
| Латентность действия | время «клик → подтверждение сервера» | ≤150 мс p95 (тот же регион) | ≤200 мс p95 глобально |
| Доступность | uptime игрового сервиса | 99.5% | 99.9% |
| Reconnect | окно переподключения без потери места | ≥120 сек | ≥120 сек |
| Нагрузка (старт) | concurrent игроков | 200–2 000 CCU на 1 узле | — |
| Нагрузка (масштаб) | concurrent игроков | — | до 1M через горизонталь |
| Трафик на партию | средний bandwidth | <5 КБ/с на игрока (пошагово, дёшево) | то же |
| Холодный старт клиента | first meaningful paint | ≤2.5 с на 4G | ≤2.5 с |
| Стоимость инфры (старт) | месячный бюджет | $20–80/мес (донаты покрывают) | растёт линейно с CCU |

> **Ключевой инженерный факт:** пошаговая игра почти не грузит сеть (событие на ход, не 60 пакетов/сек). Поэтому узкое место — **число открытых WebSocket-соединений и матчмейкинг**, а не CPU игровой логики. Это позволяет стартовать очень дёшево.

## 2. Архитектура системы (высокоуровнево)

### 2.1. Принципы
- **Authoritative server** — сервер единственный источник истины (анти-чит, синхронность, честный ГСЧ). Клиент ничего не решает, только рендерит и отправляет намерения (intents).
- **Stateful game rooms** — состояние партии живёт в памяти конкретного инстанса; комната «приклеена» к узлу (sticky by room).
- **Stateless периферия** — auth, lobby, matchmaking, presence, профили — горизонтально масштабируемы, без липкости.
- **Event sourcing** — каждое игровое действие = неизменяемое событие в логе. Бесплатно даёт: реплеи, аудит честности ГСЧ, восстановление после краша, async-режим, аналитику.
- **Детерминированное ядро правил** — игровой движок = чистая функция `(state, event) → newState`. Одинаково исполняется на сервере (авторитет) и на клиенте (предсказание/анимация). Один и тот же код — изоморфный.

### 2.2. Диаграмма компонентов (текстом)

```
                         ┌─────────────────────────┐
        Players  ───────▶│   CDN / Static (PWA)     │  клиент (Pixi.js 2.5D)
                         └─────────────────────────┘
                                     │ WSS / HTTPS
                                     ▼
                         ┌─────────────────────────┐
                         │   Edge / Load Balancer   │  (TLS, WS upgrade, sticky-by-room)
                         └─────────────────────────┘
              ┌──────────────────────┼───────────────────────┐
              ▼                      ▼                        ▼
   ┌──────────────────┐   ┌────────────────────┐   ┌────────────────────┐
   │  Gateway / API    │   │  Realtime Game     │   │  Matchmaking /     │
   │  (REST, stateless)│   │  Server (stateful) │   │  Lobby (stateless) │
   │  auth, profiles,  │   │  Colyseus-like     │   │  presence, queues  │
   │  replays, stats   │   │  game rooms        │   │                    │
   └──────────────────┘   └────────────────────┘   └────────────────────┘
              │                      │                        │
              └──────────┬──────────┴────────────┬───────────┘
                         ▼                        ▼
              ┌────────────────────┐   ┌────────────────────┐
              │  PostgreSQL        │   │  Redis             │
              │  (users, replays,  │   │  (presence, queues,│
              │   stats, seeds)    │   │   pub/sub, cache,  │
              │                    │   │   session sticky)  │
              └────────────────────┘   └────────────────────┘
                         │
                         ▼
              ┌────────────────────┐
              │  Object Storage     │  (event-логи партий, реплеи, ассеты)
              │  (S3-compatible)    │
              └────────────────────┘
```

### 2.3. Почему так (обоснование)
- **Sticky-by-room, а не sticky-by-user** — все игроки одной партии идут на один узел; межсерверная синхронизация партии не нужна (дёшево и просто).
- **Redis Pub/Sub** связывает узлы для presence/лобби/чатов и позволяет позже добавить cross-node фичи (зрители турниров) без переписывания.
- **Event sourcing в Object Storage** — лог партии пишется потоково; PostgreSQL хранит метаданные и индексы, тяжёлые логи — в дешёвом S3.

## 4. Игровое ядро: детерминированный движок правил

### 4.1. Контракт ядра
```typescript
// Чистая функция: одинаково на сервере и клиенте
function reduce(state: GameState, event: GameEvent): GameState;

// Применение намерения игрока проходит валидацию:
function validate(state: GameState, intent: PlayerIntent, playerId: PlayerId): 
  | { ok: true; events: GameEvent[] }
  | { ok: false; reason: RejectReason };
```
- **Intent** — намерение игрока («построить дорогу здесь»). Приходит с клиента.
- **Event** — факт, порождённый сервером после валидации («дорога построена», «брошены кубики: сид=…, результат=8»). Только события меняют состояние.
- **Детерминизм:** при одинаковом начальном состоянии + одинаковой последовательности событий результат идентичен на любом устройстве. Это основа реплеев, предсказания и проверки честности.

### 4.2. Честный ГСЧ (закрытие главной боли)
1. При старте партии сервер генерирует `seed` (криптослучайный) и **хеширует** его: `seedHash = SHA256(seed)`. `seedHash` показывается игрокам **до** партии (commit).
2. Все случайные события (броски, перетасовка колоды чисел, развитие) выводятся **детерминированно** из `seed` + номера события (PRNG-stream).
3. После партии публикуется **сам `seed`** (reveal). Любой игрок может пересчитать все броски из event-лога и убедиться: ничего не подкручивалось.
4. Это **commit-reveal схема** — стандарт доказуемой честности. Прямой ответ на репутационный провал Colonist.io.

### 4.3. Профили правил (Rule Profiles)
Конфиг-объект, а не разные кодовые ветки:
```typescript
interface RuleProfile {
  id: 'classic' | 'balanced' | 'blitz' | 'deep';
  randomness: 'dice' | 'balanced_deck';   // кубики или колода чисел
  catchUp: {
    friendlyRobber: boolean;               // нельзя грабить игрока с ≤2 VP
    robinHood: boolean;                    // poverty-токены отстающим
    finalRound: boolean;                   // доигрывание круга после порога
    hiddenVP: boolean;                     // часть VP скрыта до триггера
    catchUpEvents: boolean;                // часть «7» → бусты отстающим
  };
  victoryPoints: number;                   // адаптивный порог
  boardSize: 'small' | 'standard' | 'large';
  turnTimer: { soft: number; hard: number }; // сек
  parallelPhases: boolean;                 // одновременная торговля/сброс
}
```
Профили — это и есть «мульти-режимная платформа» (вводная №4): движок один, поведение конфигурируется.

### 4.4. Адаптивная длительность (потолок 60 мин)
Перед стартом lobby рассчитывает конфиг под (число игроков × режим), целясь в окно из Фазы 1:

| Режим | Игроков | Окно | VP | Поле | Таймер hard |
|---|---|---|---|---|---|
| Blitz | 2–10 | 15–25 мин | 7–8 | small | 20 с |
| Classic | 3–4 | 40–60 мин | 10 | standard | 45 с |
| Classic | 5–6 | ≤60 мин | 10 | large + parallel | 40 с |
| Deep | 3–4 | ≤60 мин | 13 | standard | 60 с |
| Async | 2–6 | вне RT | конфиг | конфиг | часы/ход |

Если расчёт рискует пробить 60 мин — движок понижает VP / ужесточает таймеры и **предупреждает в лобби**.

## 5. Сетевой протокол

### 5.1. Транспорт
- **WebSocket (WSS)** — основной канал реального времени (пошаговая игра, UDP не нужен).
- **HTTPS/REST** — всё, что не realtime: auth, профили, история, реплеи, донаты.
- Сериализация: **JSON** на старте (читаемо, легко контрибьютить) → опц. **MessagePack/бинарь** при масштабе для экономии трафика.

### 5.2. Категории сообщений (клиент → сервер)
| Тип | Назначение |
|---|---|
| `auth.connect` | установка сессии, токен |
| `room.create` / `room.join` / `room.leave` | управление комнатой |
| `intent.*` | игровые намерения (build, trade.offer, trade.accept, roll, playDevCard, moveRobber, endTurn …) |
| `chat.message` / `chat.reaction` | соц-слой |
| `ping` | keepalive/latency |

### 5.3. Категории сообщений (сервер → клиент)
| Тип | Назначение |
|---|---|
| `room.state` | полное состояние при входе/реконнекте |
| `event.batch` | пачка авторитетных событий (дельта состояния) |
| `intent.rejected` | отказ с причиной (анти-чит/валидация) |
| `turn.timer` | синхронизация таймеров |
| `chat.*` | трансляция соц-сообщений |
| `room.closed` | завершение партии + reveal сида |

### 5.4. Client prediction (отзывчивость)
- На «безопасные» действия (выбор пятна, открытие меню) клиент применяет предсказание мгновенно через то же ядро `reduce`.
- Сервер подтверждает событием; при расхождении — **reconciliation** (откат к авторитетному состоянию). Для пошаговой игры расхождения редки.

### 5.5. Reconnect + bot-fill (закрытие боли «кикнуло за 3 пропуска»)
1. Дисконнект → место держится **grace-window ≥120 с**; на ход временно ходит бот (опц.) или ход скипается мягко.
2. Реконнект в окне → `room.state` восстанавливает партию полностью (из in-memory + event-лог).
3. Окно истекло → постоянная замена ботом, партия не ломается. Игрок может вернуться зрителем.
4. **Никаких «вечных кармических банов»** за дисконнект (прямая претензия к Colonist).

## 6. Модель данных

### 6.1. PostgreSQL (основные таблицы)

```sql
-- Пользователи (OAuth + гость)
users (
  id            uuid primary key,
  username      text unique not null,
  email         text unique,            -- null для гостей
  auth_provider text,                   -- google/discord/guest
  created_at    timestamptz default now(),
  is_guest      boolean default false,
  deleted_at    timestamptz             -- self-service удаление (GDPR)
)

-- Рейтинг / прогресс
player_ratings (
  user_id   uuid references users(id),
  mode      text,                       -- ranked profile id
  rating    int default 1200,           -- Glicko-2/Elo
  rd        real,                       -- rating deviation (Glicko-2)
  season_id int,
  games     int default 0,
  wins      int default 0,
  primary key (user_id, mode, season_id)
)

-- Партии (метаданные; тяжёлый лог — в S3)
matches (
  id           uuid primary key,
  profile      jsonb not null,          -- RuleProfile конфиг
  seed_hash    text not null,           -- commit
  seed         text,                    -- reveal (после финала)
  player_count int,
  status       text,                    -- live/finished/abandoned
  started_at   timestamptz,
  finished_at  timestamptz,
  event_log_uri text,                   -- ссылка на S3 event-лог
  winner_id    uuid references users(id)
)

-- Участники партии
match_players (
  match_id  uuid references matches(id),
  user_id   uuid references users(id),
  seat      int,
  final_vp  int,
  result    text,                       -- win/loss/abandoned
  primary key (match_id, seat)
)

-- Донаты (прозрачность OSS)
donations (
  id         uuid primary key,
  user_id    uuid references users(id), -- null = аноним
  amount     numeric,
  currency   text,
  provider   text,                      -- stripe/ko-fi/opencollective
  created_at timestamptz default now(),
  public     boolean default true       -- для публичной страницы спонсоров
)
```

### 6.2. Redis (структуры в памяти)
| Ключ/структура | Назначение |
|---|---|
| `presence:{userId}` | онлайн-статус, TTL |
| `queue:{mode}` (sorted set) | очередь матчмейкинга по рейтингу |
| `room:{roomId}:node` | какой узел держит комнату (sticky-routing) |
| `pubsub: lobby/chat/presence` | межузловые события |
| `session:{token}` | сессии, реконнект-токены |

### 6.3. Object Storage (S3)
- `matches/{id}/events.ndjson` — event-лог партии (источник истины для реплея/аудита).
- `replays/{id}.json` — предрассчитанный реплей (опц., для быстрой загрузки).
- `assets/...` — спрайты, атласы, темы.

### 6.4. Event-лог (формат события)
```json
{ "seq": 42, "ts": 1719600000, "type": "DICE_ROLLED",
  "actor": "system", "data": { "result": 8, "rngStreamIndex": 17 } }
```
Из `seed` + `rngStreamIndex` любой может пересчитать `result` → доказательство честности.

## 7. API (REST, основное)

> Realtime — через WebSocket (раздел 5). REST — для всего остального. Спецификация — **OpenAPI 3.1**.

| Метод | Путь | Назначение |
|---|---|---|
| POST | `/auth/guest` | гостевая сессия (без регистрации — one-click play) |
| GET | `/auth/{provider}` | OAuth (Google/Discord) |
| GET | `/me` | профиль текущего пользователя |
| DELETE | `/me` | self-service удаление аккаунта (GDPR) |
| GET | `/matches/{id}` | метаданные партии |
| GET | `/matches/{id}/replay` | реплей (event-лог) |
| GET | `/matches/{id}/verify` | вернуть seed + позволить пересчёт честности |
| GET | `/leaderboard/{mode}` | рейтинги/лиги |
| GET | `/stats/{userId}` | пост-матч аналитика игрока |
| POST | `/rooms` | создать приватную комнату (вернуть join-код/ссылку) |
| GET | `/donations/public` | публичная страница спонсоров (прозрачность) |
| POST | `/donations/webhook` | вебхук провайдера донатов |

Матчмейкинг и присоединение к комнате инициируются по REST, но переходят в WS-сессию.

## 8. Оценка нагрузки и масштабирование

### 8.1. Профиль нагрузки (почему это дёшево)
- 1 игрок = 1 WS-соединение + ~1 событие на ход. При ходе раз в ~15–30 с трафик <5 КБ/с.
- CPU игровой логики на ход ничтожен (это не физика реального времени).
- **Узкое место №1:** количество одновременных WS-соединений на узел.
- **Узкое место №2:** матчмейкинг и presence (Redis).

### 8.2. Ёмкость (ориентиры)
| Конфигурация | Оценка CCU |
|---|---|
| 1 VPS (4 vCPU/8GB), Node+Colyseus | ~2 000–5 000 одновременных игроков (сотни параллельных партий) |
| + вертикальное усиление узла | ~10k на узел (зависит от GC/heap) |
| Горизонталь: N realtime-узлов + sticky-by-room | линейно: ~10k × N |
| Бенчмарк-потолок WS на спец-стек (для справки) | один оптимизированный узел способен держать ~1M WS-соединений — это ориентир для «маркетинговой планки», не стартовая цель |

> **Путь к 1M (когда/если понадобится):** stateless-периферия (auth/lobby/presence) масштабируется тривиально; realtime-узлы добавляются шардами комнат; Postgres → read-replicas + партиционирование; Redis → cluster; при упоре Node в соединения — вынос realtime-слоя на Elixir/Go. Всё это — **отложенные двери**, не стартовая работа (вводная №5).

### 8.3. Стартовая стратегия (реалистичный слабый старт)
- 1 узел, docker-compose, Cloudflare free, managed Postgres free-tier.
- Бюджет **$20–80/мес**, покрывается донатами.
- Автоскейл и k8s **не делаем на старте** — только закладываем границы модулей, чтобы переход был дешёвым.

### 8.4. Анти-чит и честность
- Сервер авторитетен: клиент не может «дорисовать» ресурсы/ходы (любой невалидный intent → `intent.rejected`).
- Скрытая информация (карты развития соперников) **не отправляется** на клиент до раскрытия.
- ГСЧ commit-reveal (4.2) — игрок не может предсказать/манипулировать бросками.
- Rate-limiting intents, защита от спама, серверная валидация таймеров.

## 11. Дорожная карта (milestones)

> Принцип: каждый milestone — играбельный инкремент. Мульти-режимность закладываем в ядро с M1, контент включаем профилями.

### M0 — Прототип и валидация (gate-решения)
- Прототип рендера: **Pixi.js vs Three.js** на реальном поле → решение 2.5D-движка.
- Скелет монорепо, CI, детерминизм-тест ядра.
- **Результат:** подтверждённый стек, hello-world партия локально.

### M1 — Вертикальный срез (играбельное ядро)
- `@skervik/core`: базовые правила (поле, ресурсы, строительство, торговля, разбойник, победа).
- Authoritative-сервер (Colyseus), WS-протокол, 1 комната.
- Клиент: 2.5D-поле, ходы, торговый UI (приоритет — безошибочность).
- ГСЧ commit-reveal.
- **Результат:** 3–4 игрока играют Classic-партию онлайн в одной комнате.

### M2 — Платформа режимов и устойчивость
- Профили правил (Classic/Balanced/Blitz), адаптивная длительность.
- Catch-up механики (friendly robber, final round, hidden VP).
- Reconnect + grace-window + bot-fill.
- Матчмейкинг, лобби, приватные комнаты по коду, presence (Redis).
- Боты (heuristic) для single-player и bot-fill.
- **Результат:** мульти-режимная платформа 2–6 игроков, single + multiplayer.

### M3 — Соревновательность и соц-слой
- Ranked, рейтинг (Glicko-2), сезоны, лиги, антисмурф.
- Друзья, гильдии, чат, реакции, анти-токсичность.
- Spectator mode, реплеи, пост-матч аналитика (+ закрытие паранойи по ГСЧ через данные).
- Режимы 7–10 (Blitz + командный).
- **Результат:** полноценная соревновательная платформа.

### M4 — Полировка, доступность, контент
- Async-режим (play-by-turn).
- Deep/Marathon профиль (товары/рыцари/треки — слой «городов и рыцарей»).
- Темы/скины поля, премиальный арт, кастомизация.
- a11y (дальтоник-режимы), полная локализация UA/RU/EN, мобильные обёртки (PWA + сторы).
- **Результат:** релиз 1.0.

### M5 — Масштаб (по факту спроса)
- Горизонтальный шардинг realtime, managed Postgres/Redis, k8s, автоскейл.
- Опц. вынос realtime-слоя на Elixir/Go при упоре в соединения.
- **Результат:** готовность к росту CCU без переписывания.

## 14. Что решить/уточнить перед M0

1. **Лицензия:** AGPL-3.0 (рекомендую для сетевого сервиса) vs MIT.
2. **OAuth-провайдеры на старте:** Google + Discord? (Discord близок игровой аудитории).
