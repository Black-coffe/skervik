# Skervik — план исправлений по итогам аудита (июль 2026)

## 1. Шапка

**Дата аудита:** 2026-07-02
**Коммит на момент аудита:** `2d70e48` («feat(core): event-log format + replayLog + golden determinism test (M0 E0.5 S0.5.4)»)

**Статус проекта одним абзацем.** Skervik сейчас — это M0-скаффолд, а не продакшн и не играбельный прототип. Репозиторию два дня, все коммиты от Claude Code + Dependabot — второго живого контрибьютора пока не появилось. `pnpm -r typecheck/lint/test/build` зелёные на всех пяти пакетах, CI настроен и работает. Но реальная логика есть только в одном пакете из пяти — `@skervik/core` (детерминированное ядро правил: типы, `reduce`, `validate`, сидированный PRNG, replay). Остальные четыре пакета (`protocol`, `server`, `client`, `bots`) — это заглушки: `index.ts` на 2–4 строки, экспортирующий константу вида `X_VERSION = '0.0.1'`, и один тривиальный тест, который эту константу проверяет. Игровых правил Catan-подобной механики нет вообще: ни производства ресурсов, ни построек, ни торговли, ни разбойника — только каркас хода (turn order). Рендеринга (Pixi.js) нет, сервера (Colyseus/Fastify) нет, ботов нет. Коммерческая/сообщественная часть (Discord, Open Collective, лор, арт) на нуле — решения приняты (ADR), но не реализованы физически. Это ожидаемо и нормально для стадии M0 (см. `docs/specs/roadmap/ROADMAP.md`) — документ ниже не про «всё сломано», а про то, что нужно докрутить, прежде чем M0 можно будет считать честно закрытым, и про недорогие продуктовые действия, которые можно сделать уже сейчас, не дожидаясь M1.

## 2. Что сделано хорошо (не трогать / не «улучшать»)

Исполнителю **не нужно** переделывать или облегчать перечисленное ниже — это осознанные и работающие решения:

1. **Детерминизм-гард в ESLint** — кастомное правило в `eslint.config.js`, которое запрещает `Math.random()`, `Date.now()`, `new Date()` внутри `packages/core/src/**`. Это реальный enforced-гейт архитектурного инварианта «нет ambient randomness в ядре» — основа детерминизма и честного RNG.
2. **Строгий `tsconfig.base.json`** — `strict`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `noImplicitOverride`, `noFallthroughCasesInSwitch`. Не занижать строгость ради удобства.
3. **CI-гейт полный и работает** — install (frozen lockfile) → guard на «zero runtime deps» у core (ADR-0003) → typecheck → lint → prettier --check → test → build. Плюс CodeQL и Dependabot.
4. **ADR-дисциплина** — 7 принятых ADR (лицензия, рендер-движок, детерминированное ядро, realtime-стек, auth, донаты, имя бренда), все согласованы с кодом и роадмапом, у каждого есть Context/Options/Decision/Consequences/Revisit-when. Новые архитектурные развилки — тоже через ADR, не через код без объяснения.
5. **Ядро `@skervik/core`** спроектировано и написано верно: чистые данные без классов/`Map`, `reduce()` — чистая функция с исчерпывающим `switch` и `never`-веткой по умолчанию, не мутирует вход (это проверяется тестом), PRNG (mulberry32 в counter-режиме) с честными golden-value регрессионными тестами. Основа под «provably fair RNG» уже реализована качественно — просто ещё не подключена (см. пункт A1).
6. **OSS-гигиена для 2-дневного репозитория необычно сильная** — README, CONTRIBUTING (с явным списком hard invariants и DCO), SECURITY.md с реальным приватным флоу раскрытия и контактом, CODE_OF_CONDUCT, issue/PR-шаблоны, `dependabot.yml`. Не нужно писать это заново — только точечно поправить (см. A4/A5, B1).
7. **Никакого технического долга в виде TODO/FIXME/HACK** — в коде их нет вообще; единственный сознательно помеченный плейсхолдер (A1) прокомментирован как «M0 placeholder», то есть команда сама знает об этом разрыве.

## 3. Раздел A — Технические/функциональные исправления

Пункты упорядочены по приоритету (A1 — самый важный).

### A1. Подключить сидированный PRNG к игровой логике (сейчас — плейсхолдер-формула)

**Что не так.** В `packages/core/src/validate.ts` (строка 64), в ветке обработки intent-а `intent.rollDice`, значение кубика вычисляется по формуле `2 + (state.eventIndex % 11)` — это детерминированная, но не случайная и не криптографически честная заглушка. При этом в том же пакете уже существует и полностью протестирован модуль `rng.ts` с функцией `rollDie` (сидированный PRNG на mulberry32, честные golden-тесты), но `validate.ts` его не вызывает.

**Почему это важно.** «Provably fair RNG» (commit-reveal: сервер публикует `seedHash` до матча, раскрывает `seed` после, любой может пересчитать каждый бросок из публичного лога событий) — это заявленный флагманский ответ проекта на главную репутационную проблему конкурентов (обвинения в подкрученных костях у Colonist.io, см. `docs/catan-online-research-phase.md`). Сейчас эта фича физически не работает: кубик не имеет отношения к seed-у и не может быть честно верифицирован постфактум. Если этот разрыв не закрыть до того, как поверх ляжет M1-логика (производство ресурсов на бросок и т.д.), приведётся к клубку зависимостей и переделывать станет дороже.

**Как делать.**
1. В `validate.ts` в обработке `intent.rollDice` заменить формулу-плейсхолдер на вызов `rollDie` из `rng.ts`, передав туда `seed` матча и `rngStreamIndex`, полученный из текущего состояния/индекса события (тот же принцип derive, что уже используется и протестирован в `rng.ts`).
2. Проверить, что `GameState` действительно хранит и/или прокидывает `seed`, необходимый для вызова PRNG (если поля не хватает — это не расширение скоупа, а необходимая часть той же задачи, но только минимально нужное поле, без добавления смежной функциональности).
3. Пересчитать («перезолотить») golden-фикстуры `packages/core/src/__fixtures__/golden.events.ndjson` и `golden.state.json`, потому что смена формулы броска изменит эталонные значения кубика в фикстуре.
4. Обновить/дополнить существующий тест детерминизма и тест на «кубик воспроизводим из seed + rngStreamIndex» (в духе уже существующих тестов в `rng.test.ts`), но теперь проверяя именно путь `validate.ts → rollDie`, а не только сам `rng.ts` в изоляции.

**Критерии приёмки.**
- `validate.ts` больше не содержит формулы `2 + (eventIndex % 11)` или любой другой не-PRNG формулы для броска кубика.
- Значение кубика, полученное через `intent.rollDice`, идентично значению, которое даёт прямой вызов `rollDie(seed, streamIndex)` с теми же параметрами.
- Golden-фикстуры пересчитаны и совпадают с выводом `replayLog`.
- `pnpm --filter @skervik/core test` зелёный, включая тест детерминизма/replay.
- `pnpm -r typecheck/lint/test/build` зелёные.

**Скоуп.** Только `packages/core/src/validate.ts`, при необходимости `packages/core/src/types.ts` (минимальное поле для seed, если его действительно не хватает), тестовые файлы рядом с `validate.ts`/`rng.ts`, фикстуры `__fixtures__/golden.*`. Не трогать `reduce.ts`, не добавлять игровые правила (производство ресурсов и т.п. — это M1, не сюда).

**Приоритет / сложность.** Высокий приоритет, сложность **S** (одна точечная замена + пересчёт фикстур + обновление тестов).

### A2. Вынести core-детерминизм-тест в отдельный именованный шаг/гейт CI

**Что не так.** Тест на детерминизм ядра (replay golden-фикстуры → побайтово идентичное состояние) физически существует и проходит, но выполняется как часть общего шага `test` в `.github/workflows/ci.yml`, а не как отдельный именованный шаг. Формально это покрывает story `S0.3.2` из роадмапа («core determinism test in CI»), но не по духу требования — при регрессии CI покажет обобщённое «Test failed», а не сигнал именно про детерминизм. `CONTRIBUTING.md` уже сейчас формулирует это чуть громче, чем есть на самом деле («CI includes the core determinism test» — технически верно, но подано как отдельная гарантия).

**Почему это важно.** Детерминизм ядра — это архитектурный инвариант №1 проекта (без него ломается replay, честный RNG, восстановление после краша, async-режим). Такой инвариант должен быть виден в CI как отдельная именованная проверка — чтобы (а) регрессия сразу читалась как «сломан детерминизм», а не терялась в шуме обычных unit-тестов, и (б) чтобы это соответствовало собственному заявлению проекта в `CONTRIBUTING.md` и `ROADMAP.md` (Definition of Done п. «core determinism test in CI»).

**Как делать.**
1. В `.github/workflows/ci.yml` выделить отдельный именованный job или step (например, что-то в духе «core determinism gate»), который запускает именно тест(ы) детерминизма/replay из `@skervik/core` — отдельно от общего `pnpm -r test`.
2. Обычный `pnpm -r test` можно оставить как есть (он и так их выполняет) — либо явно исключить эти тесты из общего прогона, чтобы не дублировать время выполнения; выбрать более простой вариант — не исключать, а просто добавить отдельный именованный шаг, который целенаправленно гоняет именно эти тестовые файлы.
3. Убедиться, что при намеренной порче (например, временное изменение эталонного значения в фикстуре в тестовой ветке) новый именованный шаг падает явно и понятно.

**Критерии приёмки.**
- В CI появляется шаг с понятным именем, однозначно указывающим на «детерминизм / replay ядра», отдельно видимый в списке проверок PR.
- Намеренная порча детерминизма (тестовая проверка, не коммитить в main) валит именно этот шаг, а не общий `test`.
- Остальные CI-шаги (typecheck/lint/prettier/build/zero-deps guard) не затронуты и остаются зелёными.

**Скоуп.** Только `.github/workflows/ci.yml`. Можно скорректировать формулировку в `CONTRIBUTING.md`, если она перестанет точно описывать реальность после изменения.

**Приоритет / сложность.** Средний приоритет, сложность **S**.

### A3. Переименовать `name` в корневом `package.json` (хвост ребрендинга)

**Что не так.** Корневой `package.json` до сих пор содержит `name: "archipelago"` — это добрендовый кодовое название проекта. Везде остальном (README, пакеты `@skervik/*`, домен skervik.com) используется имя Skervik. ADR-0007 закрыл переименование бренда и scope-пакетов `@skervik/*`, но явно не упомянул корневой `name` в `package.json`.

**Почему это важно.** Мелкое, но заметное несоответствие для любого, кто открывает `package.json` первым — сигнализирует небрежность именно в той части, которую ADR-0007 был призван закрыть полностью. Пакет приватный (`private: true`), риска публикации под чужим именем нет, но это последний видимый хвост ребрендинга.

**Как делать.** Заменить значение поля `name` в корневом `package.json` на согласованное с брендом (например, `skervik` или `@skervik/root` — выбрать вариант, соответствующий уже принятому в ADR-0007 паттерну именования пакетов, без придумывания нового паттерна).

**Критерии приёмки.** Значение `name` в корневом `package.json` не содержит слова "archipelago"; `pnpm install`/`pnpm -r build` продолжают работать без ошибок (переименование private-пакета верхнего уровня не должно ничего сломать, т.к. на него никто не ссылается по имени).

**Скоуп.** Только корневой `package.json`.

**Приоритет / сложность.** Низкий приоритет, сложность **S** (тривиальная правка, но стоит сделать до того, как кто-то из новых контрибьюторов это заметит).

### A4. Привести описания в `package.json` пакетов в честное соответствие с реальностью

**Что не так.** Два пакета заявляют в своих `package.json` (поле `description` и/или зависимостях по смыслу) технологии, которых там физически нет:
- `@skervik/client` описан как использующий Pixi.js v8, но зависимости `pixi.js` в `package.json` нет вообще — фактически это голый React-компонент с текстом-заглушкой.
- `@skervik/protocol` описан как имеющий «runtime validation», подразумевая zod, но зависимости `zod` нет нигде в пакете.

**Почему это важно.** Это не критично технически (движки ещё не должны быть подключены — E0.4/Pixi-прототип и zod-схемы протокола запланированы позже роадмапом), но вводит в заблуждение любого, кто читает `package.json` как источник правды о текущем состоянии пакета — в том числе будущих контрибьюторов и стейкхолдеров, оценивающих прогресс. ADR-0002 (движок рендера = Pixi.js v8) формально имеет статус «accepted», хотя собственный валидационный чекпоинт (E0.4 perf-прототип) ещё не пройден — описание пакета не должно опережать реальность больше, чем сам ADR.

**Как делать.** Для каждого из двух пакетов выбрать один из двух вариантов и применить единообразно:
- Либо убрать из `description` конкретное упоминание технологии, которая ещё не подключена, оставив более общую и честную формулировку («2.5D game client (rendering engine TBD by E0.4)», «shared protocol types (runtime validation planned)» — сформулировать в том же стиле, что и остальные описания в монорепо).
- Либо явно пометить как «planned» / «not yet wired» прямо в описании, если хочется сохранить упоминание технологии как ориентир для контрибьюторов.

Выбор между вариантами — на усмотрение исполнителя, критерий один: после правки описание не должно читаться как утверждение о том, что зависимость уже используется, если её нет в `dependencies`.

**Критерии приёмки.** Ни один `package.json` в монорепо не описывает технологию как используемую, если соответствующей зависимости нет в его `dependencies`/`devDependencies`. `pnpm -r typecheck/lint/test/build` остаются зелёными (правка не затрагивает код).

**Скоуп.** `packages/client/package.json`, `packages/protocol/package.json`, `packages/server/package.json` (описания всех пакетов монорепо проверяются по одному правилу — критерий приёмки монорепо-wide). Не трогать код, не добавлять реальные зависимости pixi.js/zod/colyseus в рамках этого пункта — это отдельные story роадмапа (E0.4, S1.5.1, E1.4).

> **Примечание по исполнению (2026-07-02).** При выполнении обнаружены два нарушения сверх названных аудитом: `server` заявлял Colyseus+Fastify без зависимостей, `client` заявлял Zustand без зависимости. Оба исправлены той же правкой. `bots` и `core` проверены — нарушений нет.

**Приоритет / сложность.** Средний приоритет, сложность **S**.

### A5. Честная маркировка пакетов-заглушек (анти-«тест-театр»)

**Что не так.** 4 из 5 пакетов монорепо (`protocol`, `server`, `client`, `bots`) на данный момент состоят из `index.ts` в 2–4 строки, экспортирующего одну константу вида `X_VERSION = '0.0.1'`, и одного теста, который проверяет, что эта константа равна самой себе. Из заявленных «41 тест проходит» в CI — 5 таких тривиальных тестов-констант, формально зелёных, но не проверяющих никакой реальной логики (потому что реальной логики там пока и нет).

**Почему это важно.** Само по себе наличие заглушек на стадии M0 — это ожидаемо и правильно (роадмап явно откладывает реальную реализацию `server`/`client`/`bots`/полноценного `protocol` на M1+). Проблема не в заглушках, а в том, что метрика «41 зелёный тест» может неверно прочитаться сторонним наблюдателем (донором, потенциальным контрибьютором, стейкхолдером) как «41 единица покрытой логики», хотя фактически это ~36 содержательных тестов ядра плюс 5 тестов-плейсхолдеров. Честная маркировка не меняет код, но снимает риск завышенного впечатления о прогрессе.

**Как делать.**
1. В корневом README (или в отдельном README каждого пакета-заглушки, если они уже существуют/создаются) добавить короткую честную пометку по каждому из 4 пакетов-заглушек — что пакет является скелетом на стадии M0/M1, реальная реализация запланирована на конкретный эпик роадмапа (например: `server` → E1.4, `client` → E0.4/E1.6, `bots` → E2.4, `protocol` → S1.5.1), а имеющийся тест — это smoke-test существования пакета, а не тест логики.
2. Не обязательно (и не нужно) удалять сами smoke-тесты — они полезны как проверка, что пакет вообще собирается и импортируется; важно только не выдавать их за нечто большее.

**Критерии приёмки.** В README (корневом или пакетов) явно и без двусмысленности указано, какие 4 пакета — заглушки на M0-стадии и на какой эпик роадмапа запланирована их реализация. Формулировки нигде в документации не создают впечатления, что «41 тест» означает 41 единицу проверенной игровой/сетевой логики.

**Скоуп.** README-файлы (корневой и/или `packages/*/README.md`, если создаются). Не трогать код и тесты.

**Приоритет / сложность.** Низкий приоритет, сложность **S**.

## 4. Раздел B — Коммерческие/продуктовые исправления

### B1. Поднять Discord-сервер

**Что не так.** README уже сейчас содержит строку в духе «Community chat: Discord (link coming soon)» — то есть проект публично пообещал канал, которого физически не существует. ADR-0005 уже выбрал Discord как один из auth-провайдеров и коммуникационный хаб сообщества — то есть решение принято, реализации нет.

**Почему это важно.** Мёртвый плейсхолдер хуже отсутствия упоминания вообще — это первый сигнал «здесь заброшено» для случайного посетителя репозитория. Стоимость поднять сервер — минуты, стоимость последствий (потерянный первый контрибьютор/донор, который зашёл и увидел мёртвую ссылку) — не восполнима задним числом.

**Как делать.** Создать Discord-сервер проекта (базовая структура каналов: general, contributors/dev, announcements — на усмотрение), вставить рабочую ссылку-приглашение в README вместо плейсхолдера «coming soon».

**Критерии приёмки.** Ссылка на Discord в README ведёт на реально существующий и доступный сервер; текста «coming soon» рядом с Discord в README больше нет.

**Скоуп.** README.md, внешний сервис Discord (не код).

**Приоритет / сложность.** Высокий приоритет (дешёвая победа, устраняет действующий публичный «долг»), сложность **S**.

### B3. Написать одностраничный лор-праймер сеттинга «архипелаг исследователей»

**Что не так.** Проект юридически обязан иметь оригинальный сеттинг/лор, отличный от Catan (см. «Legal constraint» в CLAUDE.md и `docs/catan-online-research-phase.md`, вариант A — «архипелаг исследователей»). Сейчас лор-прозы не существует вообще — ни одного текста, описывающего мир, фракции, тон, названия.

**Почему это важно.** Отсутствие лора — не эстетическая недоработка, а блокер: без базового лор-документа невозможно осмысленно называть механики, писать тексты в UI, ставить ТЗ художнику на первый концепт-арт, и невозможно провести ту самую раннюю неформальную юридическую проверку на trade dress (см. B6) — проверять то, чего не существует, нельзя. Это самый дешёвый способ снять блокировку сразу нескольких последующих направлений работы (нейминг, арт, юридика) одним документом.

**Как делать.** Написать одну страницу прозы/сеттинг-бриф: название мира и его тон, кто исследователи (фракции/культуры островов), базовая география (архипелаг — какие типы островов, ресурсы, торговые пути), 3–5 характерных именований (острова, ресурсы, валюта/товары), которые естественно лягут в игровые термины. Документ размещается в `docs/` рядом с существующими research/tech-spec документами (RU, по конвенции проекта).

**Критерии приёмки.** В `docs/` появляется файл с лор-праймером (≈1 страница), содержащий достаточно оригинальных названий и образов, чтобы (а) не пересекаться терминологией с Catan, (б) дать художнику и копирайтеру опорную точку для дальнейшей работы.

**Скоуп.** Новый файл `docs/wiki/lore-primer.md` (доменные заметки по конвенции живут в `docs/wiki/`; тот же путь использует ROADMAP-2026-H2). Не трогать код.

**Приоритет / сложность.** Средний приоритет (разблокирует B6 и будущий арт), сложность **M** (требует творческой работы, не механической правки).

### B5. Разметить good-first-issues уже сейчас

**Что не так.** Проект декларирует открытость для контрибьюторов (CONTRIBUTING.md подробный, с явными hard invariants), но ни один issue не размечен как `good-first-issue`; фактический bus factor проекта равен 1 (все коммиты — Claude Code + Dependabot).

**Почему это важно.** Роадмап откладывает «good-first-issues» до M4/S4.7.5 (около релиза), но это не требует ждать: чем раньше появятся размеченные небольшие задачи, тем раньше можно проверить, придут ли вообще внешние контрибьюторы, и тем ниже риск того, что единственный maintainer останется единственным до самого 1.0 (риск «OSS maintainer burnout» явно назван в самом роадмапе, п.8, но митигация тоже отложена на M4 — стоит сдвинуть проверку раньше, это дёшево).

**Как делать.** Выбрать 2–4 действительно небольшие, изолированные задачи (например, часть пунктов раздела A этого документа сама по себе годится — A3, A4, A5 — они малорискованные, скоуп чётко ограничен, не требуют глубокого погружения в архитектуру), создать под них GitHub issues с меткой `good-first-issue` и понятным описанием критериев приёмки.

**Критерии приёмки.** В issue-трекере репозитория есть минимум 2 issue с меткой `good-first-issue`, каждый с самодостаточным описанием (не требует контекста из этого документа или устной беседы).

**Скоуп.** GitHub issues (не код, не файлы репозитория, кроме, возможно, `.github/ISSUE_TEMPLATE` при необходимости — но шаблоны уже существуют).

**Приоритет / сложность.** Средний приоритет, сложность **S**. Зависимость: логично делать после A3–A5 (чтобы не размечать issue на то, что вы одновременно чините сами в этом же цикле).

### B6. Ранняя неформальная юридическая проверка trade dress

**Что не так.** Формальный юридический/IP-ревью запланирован на `S4.7.3`, то есть в самом конце M4 (недели 32–44) — **после** первого настоящего арт-пасса (`E4.3`, тоже M4). Это означает, что если визуальный стиль или конкретные образы случайно окажутся слишком близки к узнаваемому trade dress Catan (характерные текстуры плиток ресурсов, специфичные иконки портов и т.п.), об этом станет известно только тогда, когда разворачивать арт-направление уже дорого.

**Почему это важно.** Юридическая независимость от бренда CATAN — это зафиксированный в CLAUDE.md non-negotiable констрейнт (не название, не арт, не конкретные тексты). Формальный ревью перед самым релизом — это нормально и достаточно как финальная проверка, но полагаться только на него — рискованно именно потому, что первая же серьёзная арт-работа (`E4.3`) предшествует ему на месяцы. Дешевле «спросить рано и один раз неформально», чем «переделывать арт-направление после того как формальный ревью найдёт проблему».

**Как делать.** Это процессный, не кодовый пункт: как только появится первый настоящий концепт-арт (даже один эскиз доски/тайла/иконки ресурса — не обязательно дожидаться полного `E4.3`), провести быструю неформальную сверку с юристом/консультантом (или хотя бы структурированный самостоятельный чек-лист по общедоступным признакам trade dress) — раньше, чем визуальный стиль будет закреплён и растиражирован по всем ассетам. Зафиксировать этот шаг как явный чек-поинт в самом роадмапе (например, примечанием в `docs/specs/roadmap/ROADMAP.md` у эпика E4.3 — «неформальная IP-сверка на первом эскизе, до масштабирования арт-стиля»), не дожидаясь S4.7.3.

**Критерии приёмки.** В роадмапе (или в отдельном ADR/wiki-заметке) явно зафиксирован чек-поинт «неформальная IP/trade-dress проверка на первом концепт-арте, до E4.3 масштабирования» — как дополнение к уже существующему формальному S4.7.3, а не замена ему.

**Скоуп.** Документация (`docs/specs/roadmap/ROADMAP.md` — точечное примечание у E4.3, либо новая заметка в `docs/wiki/`). Не код. Не переносить сам S4.7.3 — он остаётся финальным формальным гейтом.

**Приоритет / сложность.** Средний приоритет (сработает только в будущем, но зафиксировать чек-поинт нужно сейчас, пока не забыли), сложность **S** — это правка документа, а не выполнение самой проверки прямо сейчас (проверку выполнять, когда появится арт).

### B7. Снять противоречие «a11y/i18n baked in, not retrofitted» vs факт, что всё это в M4

**Что не так.** CLAUDE.md и роадмап декларируют принцип: «Accessibility from the start — resources must be distinguishable by more than color (colorblind modes); i18n (UA/RU/EN) baked in, not retrofitted». При этом фактическая реализация (`E4.4` — цветослепые палитры, масштабирование шрифта, i18n-фреймворк) запланирована только на M4, то есть почти в самом конце. Это не ошибка планирования (архитектурно закладывать поддержку с нуля дешевле, чем интегрировать её задним числом в готовый UI — сама идея верная), но текущий роадмап не фиксирует, *что именно* должно быть архитектурно заложено уже в M1, чтобы формулировка «baked in» не осталась пустой декларацией.

**Почему это важно.** Если в M1 UI-код будет писаться с хардкод-строками на английском и с цветом как единственным индикатором ресурса/состояния (что типично для «сделаем МVP быстро, а i18n/a11y добавим потом»), то к M4 «retrofit» неизбежен несмотря на декларацию — потому что перевести хардкод-строки в ключи локализации и добавить не-цветовые индикаторы поверх уже написанного UI кода — это ровно тот retrofit, который принцип обещал не делать. Нужно не откладывать реализацию (это разумно), а зафиксировать минимальный архитектурный чек-лист для M1, чтобы UI с самого первого рендера физически поддерживал будущую интеграцию без переписывания.

**Как делать.** Добавить в story-файлы или в сам роадмап (в эпик `E1.6` — клиентский рендер/HUD, M1) явный архитектурный чек-лист, обязательный уже на этой стадии, даже если сама реализация переводов/палитр — M4:
- Весь пользовательский текст в UI-компонентах идёт через ключи локализации/строковые константы в одном месте (не хардкод строк прямо в JSX/шаблонах), даже если сейчас заполнен только английский или один язык.
- Ресурсы/состояния, которые сейчас различаются только цветом (например, типы ресурсов на доске, статусы игроков), с первого же рендера сопровождаются вторым визуальным признаком — иконкой, паттерном или текстовой меткой, а не только оттенком.

Эти два требования — не про качество перевода или финальные цветослепые палитры (это по-прежнему M4), а про то, чтобы структура кода с первого дня не создавала долг, который M4 не сможет закрыть без переписывания.

**Критерии приёмки.** В `docs/specs/roadmap/ROADMAP.md` (у эпика E1.6) или в отдельном `docs/wiki/`-документе явно зафиксирован чек-лист из двух пунктов выше как обязательное архитектурное требование к любому UI-коду начиная с M1, а не только с M4.

**Скоуп.** Документация (роадмап или `docs/wiki/`). Не код — в M1 ещё не написан клиентский UI, эта задача только фиксирует требование к будущей реализации, чтобы её не забыли на старте M1.

**Приоритет / сложность.** Средний приоритет (важно зафиксировать до старта M1/E1.6, иначе поздно), сложность **S**.

## 5. Раздел C — Инструкция исполнителю

Этот план предназначен для выполнения по одному пункту за раз — Sonnet-моделью или мидл-разработчиком, без необходимости держать в голове весь документ целиком.

1. **Один пункт — один цикл работы.** Не смешивать несколько пунктов (даже связанных) в одном заходе. Каждый пункт (A1, A2, … B7) закрывается независимо.
2. **Тир-маршрутизация по CLAUDE.md.** Перед началом работы над пунктом классифицировать его по таблице сложности из `CLAUDE.vulyk.md` (Tier 0–4) и следовать соответствующему протоколу. Большинство технических пунктов (A1–A5) — Tier 1–2 (один модуль/фича внутри модуля); большинство продуктовых пунктов (B1–B7) — Tier 0–1 (правка документа/внешнего сервиса).
3. **Не выходить за скоуп пункта.** Каждый пункт явно перечисляет, какие файлы/области можно трогать («Скоуп» в описании). Если для решения пункта требуется затронуть что-то за пределами этого списка — остановиться и сообщить об этом, а не расширять правку молча (Закон 3 из `CLAUDE.vulyk.md`: «No out-of-scope edits»).
4. **Отдельная ветка и отдельный коммит на каждый пункт**, conventional-commits формат (как в существующей истории репозитория — например `fix(core): wire seeded PRNG into validate.ts` для A1, `docs(readme): add discord invite link` для B1 и т.д.). Не объединять несколько пунктов в один коммит.
5. **CI должен быть зелёным перед тем, как считать пункт закрытым** — `pnpm -r typecheck`, `pnpm -r lint`, `pnpm -r test`, `pnpm -r build` (там, где применимо к пункту; для чисто документационных/внешних-сервисных пунктов из раздела B техническая CI-проверка может быть не нужна, но не должна быть сломана).
6. **Критерии приёмки — это чек-лист, не пожелание.** Пункт считается закрытым только тогда, когда все перечисленные в его описании критерии приёмки выполнены, а не когда «примерно похоже сделано».
7. **Нерешаемое — фиксировать и останавливаться, не импровизировать.** Если в процессе пункта выясняется, что описанный в этом документе подход не работает (например, для A1 в `GameState` действительно физически негде хранить `seed` без более широкого рефакторинга типов) — не выдумывать более широкое решение на месте. Зафиксировать находку (что именно не сходится и почему) и остановиться для решения на уровне планирования, а не тихо расширять скоуп.
8. **Порядок выполнения не строго фиксирован**, кроме зависимостей, явно указанных в сводной таблице ниже (раздел 6) — можно начинать с самого дешёвого/безопасного пункта, чтобы наработать уверенность (например, A3 или B1), и двигаться к более сложным (A1, B3).
