# S0.5.3 — Seeded PRNG + stream-index derivation

## Goal
A pure, **zero-dependency** seeded PRNG and a deterministic way to derive a random
value from `(seed, streamIndex)` — the math foundation of the commit-reveal fair-RNG
(`docs/wiki/fair-rng-commit-reveal.md`). Reproducible across runs and machines.

## Files
- `packages/core/src/rng.ts` (PRNG + derivation helpers)
- `packages/core/src/index.ts` (re-export)
- `packages/core/src/rng.test.ts`

## Map slice
`docs/wiki/fair-rng-commit-reveal.md`, ADR-0003.

## Acceptance criteria
- [ ] PRNG implemented **in-house** (e.g. mulberry32 / xoshiro128**), no runtime dep added to core.
- [ ] `deriveValue(seed, streamIndex)` is deterministic and **independent of call order** (indexing into the stream, not stateful global draws).
- [ ] Helpers return integers where used for game randomness (e.g. `rollDie → 1..6`, `shuffle`) to avoid float drift.
- [ ] Tests assert **fixed golden outputs** for a known seed (so a future change to the algorithm is caught), and that the same seed reproduces the same sequence.
- [ ] No nondeterminism (guard enforces); zero runtime deps.

> Note: `seedHash = SHA256(seed)` (the commit step) is a **server** concern (M1 S1.4.3),
> not core — core stays crypto-free/isomorphic and only derives values from the seed.

## Implementation notes

- **Files:** added `packages/core/src/rng.ts`, `packages/core/src/rng.test.ts`;
  additively re-exported `Seed`, `deriveValue`, `rollDie`, `shuffle` from
  `packages/core/src/index.ts` (kept existing `reduce`/`replay`/`validate`/types exports).
- **Algorithm:** mulberry32 used in *counter* mode, not chained-generator mode —
  each `deriveValue(seed, streamIndex)` call (1) FNV-1a hashes the seed string to
  a uint32, (2) mixes in `streamIndex` via a Weyl-sequence-style multiply-XOR
  (`0x9e3779b9`) to spread adjacent indices apart, (3) runs one mulberry32
  scramble round on the combined state. No shared mutable generator state
  between calls, so `deriveValue` is trivially order-independent (each call
  is a pure function of `(seed, streamIndex)` alone) — verified by a test that
  draws indices 0..4 before drawing index 5 and asserts the result is unchanged.
- **Helpers:** `rollDie(seed, streamIndex) => 1..6` (`1 + floor(deriveValue * 6)`);
  `shuffle(items, seed, streamIndex)` is a non-mutating Fisher-Yates that consumes
  one `deriveValue` stream slot per swap (`streamIndex .. streamIndex + n - 2`).
- **Golden seed:** `'skervik-golden-seed-1'`. Example fixed outputs asserted in
  tests: `deriveValue(seed, 0) === 0.62726592971012`,
  `deriveValue(seed, 5) === 0.11778039508499205`,
  `rollDie` for indices 0..14 `=== [4,5,1,5,6,1,4,4,1,1,3,4,3,2,3]`,
  `shuffle([1..8], seed, 10) === [1,8,6,7,2,3,5,4]`.
- **Decision:** chose the counter-based (hash-and-scramble) construction over a
  classic stateful mulberry32 generator specifically to satisfy the
  order-independence acceptance criterion in O(1) per draw, at the cost of one
  extra FNV-1a hash pass per call (negligible; seed strings are short).
- **Not done (intentionally, per story scope):** `validate.ts`'s `intent.rollDice`
  still uses its M0 placeholder formula — wiring `rollDie` into rule logic is
  explicitly an M1 concern per the existing comment in `validate.ts`, not this story.
- Verified: `pnpm --filter @skervik/core test` (32 passed), `pnpm -r typecheck`,
  `pnpm --filter @skervik/core lint`, `pnpm exec prettier --check packages/core/src`,
  `node scripts/check-core-no-runtime-deps.mjs` — all green.
