# S0.5.1 — Core domain types

## Goal
`@skervik/core` exports the foundational domain types that form the engine contract:
`GameState`, `GameEvent`, `PlayerIntent`, `RejectReason` (+ supporting ids/unions).
**M0 skeleton, not full rules** — minimal but real shapes that the M1 Classic rules
will extend. Also proves the workspace+alias wiring end-to-end (E0.2 review nit #3).

## Files
- `packages/core/src/types.ts` (the domain types)
- `packages/core/src/index.ts` (re-export the public surface)
- `packages/core/src/types.test.ts`
- `packages/protocol/src/index.ts` + `packages/protocol/src/index.test.ts` (add a **type-only** import of a `@skervik/core` type to prove cross-package resolution)

## Map slice
ADR-0003 + `docs/wiki/deterministic-core.md` (the contract). Core engine §4.1 of the tech spec.

## Acceptance criteria
- [ ] `GameEvent` and `PlayerIntent` are **discriminated unions** keyed by a `type` field; `RejectReason` is an enumerated union.
- [ ] `GameState` is a plain serializable object (no class instances, no functions) — so it can be event-sourced and deep-compared.
- [ ] Types are exported from `@skervik/core` and documented with brief TSDoc.
- [ ] `@skervik/protocol` imports at least one `@skervik/core` type (type-only) and its test references it — proving the workspace wiring (nit #3).
- [ ] `pnpm -r typecheck` and `pnpm --filter @skervik/core test` green; core still has **zero runtime deps**.

## Verification
`pnpm -r typecheck && pnpm --filter @skervik/core test && pnpm --filter @skervik/protocol test`

## Implementation notes
- `packages/core/src/types.ts` (new): `MatchId`, `PlayerId`, `ResourceType` (left abstract — concrete archipelago resource set is an M1 decision), `GamePhase`, `PlayerState`, `GameState` (plain, all-`readonly`, JSON-serializable), `MatchStartedEvent`/`DiceRolledEvent`/`TurnEndedEvent` → `GameEvent` (discriminated union on `type`), `RollDiceIntent`/`EndTurnIntent` → `PlayerIntent` (discriminated union on `type`), `RejectReason` (4-member string-literal union). Each export has brief TSDoc tying it back to ADR-0003/the fair-RNG wiki.
- `packages/core/src/index.ts`: added `export type * from './types.js';` alongside the existing `CORE_VERSION`.
- `packages/core/src/types.test.ts` (new): builds one of each event/intent variant, switches over `type` with a `never`-exhaustiveness check in the `default` branch, and asserts `GameState`/`GameEvent[]` round-trip unchanged through `JSON.parse(JSON.stringify(...))` (proves "plain serializable, no functions/class instances").
- `packages/protocol/src/index.ts`: added `import type { PlayerId } from '@skervik/core';` and re-exported `ProtocolPlayerId = PlayerId` as the nit #3 smoke type.
- `packages/protocol/src/index.test.ts`: added a test that assigns a string literal to `ProtocolPlayerId` and asserts `typeof` — exercises the cross-package type at both compile time (the import) and runtime (the assertion).
- **Out-of-list fix required:** `packages/protocol/tsconfig.json` had an explicit `rootDir: "src"`, which made `tsc` reject the `@skervik/core` path-alias import with `TS6059` (core's `src/*.ts` falls outside protocol's declared `rootDir`) — this is exactly the gap nit #3 exists to surface. Removed only the `rootDir` line (kept `outDir`); harmless under `tsc --noEmit` (no files are written) and `pnpm -r build` (tsup ignores `rootDir`/`outDir` from `tsconfig.json`, using its own `tsup.config.ts` `outDir`) — verified both `pnpm -r typecheck` and `pnpm -r build` green afterward. `core`/`bots`/`server`/`client` tsconfigs were left untouched (no cross-package import currently exercises them).
- Decisions: events/intents kept to a minimal 3+2 set (`match.started`/`dice.rolled`/`turn.ended`, `intent.rollDice`/`intent.endTurn`) — enough to prove the discriminated-union pattern without inventing M1 ruleset content (build/trade/robber are explicitly deferred). `ValidateResult`/`reduce`/`validate` signatures intentionally NOT added here — they belong to S0.5.2 per `docs/specs/m0-foundation/plan.md`.
