# S0.5.1 — Core domain types

## Goal
`@skervik/core` exports the foundational domain types that form the engine contract:
`GameState`, `GameEvent`, `PlayerIntent`, `RejectReason` (+ supporting ids/unions).
**M0 skeleton, not full rules** — minimal but real shapes that the M1 Classic rules
will extend. Also proves the workspace+alias wiring end-to-end (E0.2 review nit #3).

## Files
- `packages/core/src/types.ts` (the domain types)
- `packages/core/src/index.ts` (re-export the public surface)
- `packages/core/src/types.test.ts`
- `packages/protocol/src/index.ts` + `packages/protocol/src/index.test.ts` (add a **type-only** import of a `@skervik/core` type to prove cross-package resolution)

## Map slice
ADR-0003 + `docs/wiki/deterministic-core.md` (the contract). Core engine §4.1 of the tech spec.

## Acceptance criteria
- [ ] `GameEvent` and `PlayerIntent` are **discriminated unions** keyed by a `type` field; `RejectReason` is an enumerated union.
- [ ] `GameState` is a plain serializable object (no class instances, no functions) — so it can be event-sourced and deep-compared.
- [ ] Types are exported from `@skervik/core` and documented with brief TSDoc.
- [ ] `@skervik/protocol` imports at least one `@skervik/core` type (type-only) and its test references it — proving the workspace wiring (nit #3).
- [ ] `pnpm -r typecheck` and `pnpm --filter @skervik/core test` green; core still has **zero runtime deps**.

## Verification
`pnpm -r typecheck && pnpm --filter @skervik/core test && pnpm --filter @skervik/protocol test`
