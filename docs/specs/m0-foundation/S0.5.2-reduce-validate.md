# S0.5.2 — `reduce` / `validate` contract + determinism scaffold

## Goal
The two pure functions at the heart of the engine, with a minimal deterministic
implementation and a replay scaffold ready for the golden fixture (S0.5.4).

## Files
- `packages/core/src/reduce.ts` — `reduce(state, event) => newState`
- `packages/core/src/validate.ts` — `validate(state, intent, playerId) => { ok:true; events } | { ok:false; reason }`
- `packages/core/src/index.ts` (re-export)
- `packages/core/src/reduce.test.ts`

## Map slice
ADR-0003 + `docs/wiki/deterministic-core.md`. Signatures per tech spec §4.1.

## Acceptance criteria
- [ ] Signatures exactly match the contract (S0.5.1 types in, new state / events out).
- [ ] `reduce` is **pure & deterministic**: no `Date.now`/`Math.random`/I/O (ESLint guard enforces); returns a new state, never mutates the input.
- [ ] `validate` returns the discriminated `{ok}` result; an illegal intent yields `{ ok:false, reason }` (a `RejectReason`), never throws for expected rejections.
- [ ] At least one minimal end-to-end path works (e.g. a simple deterministic event applied via `reduce`, and one intent validated to events).
- [ ] A `replay(initialState, events)` helper exists (fold of `reduce`) for S0.5.4.
- [ ] `pnpm --filter @skervik/core test` green; zero runtime deps.

## Implementation notes
- `packages/core/src/reduce.ts` (new): `reduce(state, event)` — exhaustive switch on `GameEvent['type']`, every branch returns a fresh object via `{ ...state, ... }` (never assigns into `state`). `match.started` builds `players` from `playerIds` (default `victoryPoints: 0`, empty `resources`), sets `phase: 'setup'`, `turn: 1`, `currentPlayerId` to the first player. `dice.rolled` has no dedicated `GameState` field yet (M1 adds resource production) so it only advances `eventIndex` — still a real state change, satisfying the "event applied via reduce, state actually changes" criterion. `turn.ended` advances `currentPlayerId` and `turn`. Every branch sets `eventIndex: event.index + 1` (eventIndex = count of events applied so far, doubling as the next event's stream index per `types.ts`). Unreachable `default` throws on a `never`-exhaustiveness violation (same pattern as `types.test.ts`) — not an "expected rejection" path, so this doesn't conflict with validate's no-throw contract.
- `packages/core/src/reduce.ts` also exports `replay(initialState, events)` — `events.reduce(reduce, initialState)`, the fold S0.5.4's golden-fixture test will call.
- `packages/core/src/validate.ts` (new): `validate(state, intent, playerId)` returns the new `ValidateResult` type (`{ ok:true; events } | { ok:false; reason }`), defined here per the story note since it doesn't belong in S0.5.1's `types.ts`. Check order: `MALFORMED_INTENT` (intent.playerId vs. the server-authenticated `playerId` param — catches a spoofed payload even though the caller already knows the true actor) → `UNKNOWN_PLAYER` (not in `state.players`) → `INVALID_PHASE` (only `'main'` allows `rollDice`/`endTurn` in this M0 skeleton) → `NOT_YOUR_TURN`. Never throws on these paths; only the impossible `default` (unknown intent type) throws, mirroring `reduce`.
- `intent.rollDice` → one `DiceRolledEvent`. **Tradeoff surfaced:** S0.5.3 (seeded PRNG) and its wiring into rule logic are out of scope here and aren't named in any M0 story's file list, so the dice `value` is a deterministic placeholder (`2 + (state.eventIndex % 11)`) — a pure function of state, no `Math.random`/ambient draw, satisfying ADR-0003 today and replaceable (not removable) once a future story wires `rng.ts` into game logic.
- `intent.endTurn` → one `TurnEndedEvent`, `nextPlayerId` computed by rotating `state.players` (round-robin via `findIndex` + modulo). Guarded the `noUncheckedIndexedAccess` `players[...]` lookup with an explicit (practically unreachable, since membership was already checked) `undefined` → reject branch instead of a non-null assertion, keeping the "never throw for expected rejection" contract literal.
- `packages/core/src/index.ts`: added `export { reduce, replay } from './reduce.js'` and `export { validate } from './validate.js'` / `export type { ValidateResult } from './validate.js'`, ordered per `simple-import-sort/exports` (auto-fixed).
- `packages/core/src/reduce.test.ts` (new, per the story's file list — no separate `validate.test.ts`): covers `reduce` immutability (deep-clone-before/after) and all three event branches; `validate`'s four rejection reasons plus a "never throws" assertion; both intents validated end-to-end (events applied via `reduce`, state actually changes; `rollDice` re-validated against the same state to confirm the placeholder value is deterministic); `replay` folding a 3-event log equivalently to manual `reduce` chaining, plus a no-mutation check.
- Verification: `pnpm --filter @skervik/core test` (3 files / 20 tests, all green), `pnpm -r typecheck` (green, all 5 packages), `pnpm --filter @skervik/core lint` (clean after `eslint --fix` sorted the new exports), `pnpm exec prettier --check packages/core/src` (clean after `prettier --write` on `reduce.ts`), `node scripts/check-core-no-runtime-deps.mjs` (zero runtime deps confirmed).
